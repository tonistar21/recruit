# Архітектурні рішення

## ADR-001: Монорепозиторій

Використовується структура `frontend/`, `backend/`, `storage/`, `docs/`. Початкова реалізація збережена в `legacy/`, оскільки Git-репозиторій у вихідному каталозі був відсутній.

## ADR-002: Автентифікація

Access і refresh токени передаються тільки HttpOnly cookies. Для змінювальних запитів застосовується double-submit CSRF token. Refresh-сесії зберігаються сервером і можуть бути відкликані.

## ADR-003: Backend

FastAPI, SQLAlchemy 2, Alembic і PostgreSQL. У тестах дозволено SQLite для швидкої ізольованої перевірки сервісів; runtime Docker використовує PostgreSQL.

## ADR-004: Файли

Фізичні файли отримують UUID-імена у приватному volume. Метадані зберігаються в БД, завантаження та скачування проходять лише через авторизовані endpoints.

## ADR-005: PDF

Початково надається друкована HTML-версія та CSV. Вони не називаються серверною PDF-генерацією. Окремий PDF-движок не додається без потреби.

## ADR-006: Rate limiting

Для однієї development-репліки використано thread-safe in-memory limiter за scope, IP та login identifier. Заголовки proxy приймаються лише від loopback/private Docker-адрес. Для multi-instance production потрібен спільний Redis limiter.

## ADR-007: Налаштування

Змінні параметри зберігаються у `system_settings`. Ліміт upload читається під час кожного запиту. Параметри, що потребують перевипуску вже чинного токена, застосовуються до нових сесій після подальшого підключення dynamic token policy.
